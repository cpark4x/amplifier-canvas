# Act 3 — Canvas Knows What You've Built

> **Execution:** Each phase gets its own implementation plan. This document is the design spec.

**Goal:** Canvas transforms from a session viewer into a project memory system. Work across parallel sessions without conflicts, stay informed without interruption, accumulate project-level knowledge that persists.

**Tech Stack:** Electron 36 + React 19 + TypeScript + Zustand + better-sqlite3 + Playwright E2E

**CSS Oracle:** `canvas.html` — all dimension/style values come from there.

---

## Phasing

Four phases, ordered by dependency (not narrative sequence):

| Phase | Name | Scenes | Delivers |
|-------|------|--------|----------|
| 1 | Session Awareness | 3.5, 3.2, 3.8 | Sessions complete, notify, persist, browsable as history |
| 2 | Session Intelligence | 3.3, 3.4 | AI analysis tab + git/PR changes tab per session |
| 3 | Parallel Sessions | 3.1 | Multi-session with git worktree isolation |
| 4 | Project Memory | 3.6, 3.7 | Cross-session aggregation, AI assessment, stats |

Dependencies: Phase 2 needs Phase 1 data. Phase 3 is independent but benefits from Phase 1 UI. Phase 4 requires Phases 1-2 for meaningful aggregation.

## Phase 1 — Session Awareness (Scenes 3.5, 3.2, 3.8)

### Goal

Canvas tracks session lifecycle end-to-end. Sessions complete, notify, persist, and are browsable as history.

### Data Model Changes

**`src/shared/types.ts` — SessionState gains:**

```typescript
export interface SessionState {
  // Existing
  id: string
  projectSlug: string
  projectName: string
  status: SessionStatus        // 'running' | 'needs_input' | 'done' | 'failed' | 'active'
  startedAt: string
  startedBy: 'canvas' | 'external'
  byteOffset: number
  recentFiles: FileActivity[]
  workDir?: string

  // New (Phase 1)
  endedAt?: string             // ISO timestamp — set on PTY exit or session:end event
  exitCode?: number            // PTY exit code (0 = clean, non-zero = error)
  title?: string               // Auto-derived from first user prompt, ~60 chars
  promptCount?: number         // User prompts in this session
  toolCallCount?: number       // Tool invocations
  filesChangedCount?: number   // Distinct files written/edited/created
}
```

No changes to `SessionStatus` — the existing `'done'` and `'failed'` states cover completion.

**`src/main/db.ts` — Schema migration:**

```sql
ALTER TABLE sessions ADD COLUMN title TEXT;
ALTER TABLE sessions ADD COLUMN exitCode INTEGER;
ALTER TABLE sessions ADD COLUMN firstPrompt TEXT;
ALTER TABLE sessions ADD COLUMN promptCount INTEGER DEFAULT 0;
ALTER TABLE sessions ADD COLUMN toolCallCount INTEGER DEFAULT 0;
ALTER TABLE sessions ADD COLUMN filesChangedCount INTEGER DEFAULT 0;
```

The `endedAt` column already exists in the schema. No migration needed for it.

### Event Parser Additions

**`src/main/events-parser.ts` — new exports:**

```typescript
// First user message — source for auto-derived session title
extractFirstPrompt(events: ParsedEvent[]): string | undefined

// Mechanical session stats (no AI)
extractSessionStats(events: ParsedEvent[]): {
  promptCount: number
  toolCallCount: number
  filesChanged: Set<string>
  lastEventTimestamp: string
}

// Truncate first prompt to ~60 chars at word boundary, strip markdown
deriveSessionTitle(firstPrompt: string): string
```

### Session Completion Flow (Scene 3.5)

PTY exits → main process finalizes session → renderer updates.

1. **`pty.ts`** — PTY `onExit(exitCode)` fires
2. Main process reads final events from events.jsonl via `tailReadEvents()`
3. Extracts: `firstPrompt`, session stats, derives `title`
4. Writes to DB: `endedAt` (now), `exitCode`, `title`, `promptCount`, `toolCallCount`, `filesChangedCount`, `status`
5. Status: `exitCode === 0 → 'done'`, `exitCode !== 0 → 'failed'`
6. Pushes updated `SessionState[]` via existing `state:sessions-changed` IPC

**External sessions** (not started by Canvas): watcher detects `session:end` event in events.jsonl → same extraction + DB update → same IPC push.

**Sidebar effect:** Session item transitions from amber dot (running) to emerald dot (done) or red dot (failed). A "+ New session" link appears in the active section for that project.

**Terminal hint:** On PTY exit, the terminal shows the shell prompt. Amplifier itself prints resume instructions.

### Toast Notification (Scene 3.2)

When a **non-selected** session completes, a toast slides in.

**New component: `src/renderer/src/components/Toast.tsx`**

- Position: bottom-right, above terminal zone
- Content: "[title] completed" with "Review" action
- Auto-dismiss: 5 seconds, fade out
- Custom, no library — one notification type doesn't justify a dependency
- Matches design language: warm paper white bg, warm charcoal text, amber "Review" action

**Detection logic:** `setSessions` in the store diffs incoming array against current state. If any session transitions from `running`/`active`/`needs_input` → `done`/`failed`, AND that session is not `selectedSessionId`, push a toast.

**Store additions:**

```typescript
interface Toast {
  id: string
  sessionId: string
  message: string
  action?: { label: string; onClick: () => void }
}

// CanvasStore gains:
toasts: Toast[]
addToast: (toast: Omit<Toast, 'id'>) => void
dismissToast: (id: string) => void
```

### Session History (Scene 3.8)

Every completed session is browsable in the sidebar with title, stats, and resume action.

**UI location:** Sidebar, below active sessions per project.

**Layout per history entry:**

```
[dot] Session title                          2h ago
      34m · 8 prompts · 12 files          [Resume →]
```

- **Status dot:** Emerald (done), red (failed)
- **Title:** `SessionState.title` (auto-derived from first prompt)
- **Relative time:** Computed from `endedAt` — "2h ago", "yesterday", "3 days ago"
- **Stats:** Duration (`endedAt - startedAt`), `promptCount`, `filesChangedCount`
- **Resume:** Launches `amplifier resume <sessionId>` in a new PTY

**No new store state needed.** History is derived from the sessions array: filter by `status === 'done' || status === 'failed'`, sort by `endedAt` descending.

**Resume IPC — new invoke/handle channel:**

```typescript
// Add to IPC_CHANNELS:
SESSION_RESUME: 'session:resume'

// Renderer → Main (invoke):
// Request: { sessionId: string }
// Response: { success: boolean, newSessionId?: string }
```

Handler in `ipc.ts`:
1. Look up session in DB → find projectSlug → project path
2. Spawn new PTY: `amplifier resume <sessionId>` in project's workDir
3. Register as new session in DB (startedBy: 'canvas')
4. Return new session ID → renderer selects it

### Sidebar Layout

```
▸ Canvas-App
    ● main                    running · 12m
    ● dark-mode              running · 34m
    + New session
  
  HISTORY
    ✓ Auth module             2h ago · 1h 12m
    ✓ Setup scaffold          yesterday · 28m
    ✗ Failed migration        3 days ago · 4m
```

Active sessions: amber dots (running/needs_input). Completed sessions below a HISTORY divider.

"+ New session" appears when at least one session has completed for that project.

### IPC Changes

| Channel | Direction | New? | Purpose |
|---------|-----------|------|---------|
| `state:sessions-changed` | Main → Renderer | Existing | Now includes endedAt, title, stats |
| `terminal:exit` | Main → Renderer | Existing | Already fires on PTY exit |
| `session:resume` | Renderer → Main | **New** | Resume a completed session |

No new push channel for completion — the existing `state:sessions-changed` carries the update, and the store detects transitions by diffing.

## Architecture Decisions

### 1. No new IPC channel for completion detection

The existing `state:sessions-changed` push delivers the full session array. The renderer diffs to detect transitions. Avoids IPC proliferation while leveraging the existing pattern.

### 2. Mechanical summaries in Phase 1, AI summaries in Phase 2

Phase 1 derives title from first prompt (string truncation) and stats from event counts. No LLM dependency. Phase 2 upgrades to AI-generated summaries via the ANALYSIS tab pipeline. Phase 1 ships fast; Phase 2 makes it smart.

### 3. Custom Toast over library

One notification type (session completion) doesn't justify a dependency. Custom component: ~50 lines, zero dependencies, exact match to design language.

### 4. History in sidebar, not a separate view

Session history lives in the sidebar per-project, not as a standalone page. Keeps history accessible without a navigation mode switch. Phase 4 adds a richer HISTORY tab in the project-level view.

### 5. Resume via CLI delegation

`amplifier resume <sessionId>` is the canonical resume path. Canvas delegates to the CLI rather than implementing session resume. Canvas is a display layer, not a competing implementation. Same principle as Decision 5 in the STORYBOARD (CLI-first commit workflow).

### 6. Phase ordering: awareness before intelligence before parallelism

Build the single-session pipeline first (Phases 1-2), then generalize to multi-session (Phase 3). Retrofitting intelligence for N sessions is easier when it works for 1. Project memory (Phase 4) comes last because it aggregates data from all prior phases.
