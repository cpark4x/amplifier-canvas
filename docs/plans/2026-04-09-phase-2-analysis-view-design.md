# Phase 2: ANALYSIS View Design

## Goal

Build the ANALYSIS view for completed sessions — a unified panel combining mechanical session data with AI-curated sections that adapt based on what the session was about. Implements Storyboard Scenes 3.3 and 3.4, merging the original ANALYSIS and CHANGES tab concepts into one unified view.

## Background

Phase 1 shipped session lifecycle, sidebar with history, toast notifications, and session enrichment (title, stats, resume). When you click a completed session today, there's a PREVIEW placeholder in the right panel. Phase 2 replaces that with the ANALYSIS view — the structured summary that makes completed sessions meaningful.

The original Storyboard had separate ANALYSIS (Scene 3.3) and CHANGES (Scene 3.4) tabs. During design, we merged them into one unified view because the LLM curates both, making the boundary artificial. A planning session has no git changes; a coding session's changes ARE part of the analysis. One scrollable page, not two tabs.

## Key Design Decisions

1. **One unified ANALYSIS view, not two tabs.** The LLM decides which sections appear based on session type. A planning session gets `decisions` and `open-questions`. A coding session gets `changes` and `key-moments`. The boundary between "analysis" and "changes" was artificial.

2. **LLM calls go through Amplifier's existing provider infrastructure.** Canvas shells out to Amplifier as a subprocess. Canvas never touches LLM APIs, provider configs, or API keys directly. This keeps Canvas provider-agnostic.

3. **Mechanical parts render instantly; AI parts load async on first view.** The tab is useful immediately (stats, prompt history), and AI sections fill in seconds later.

4. **On-demand generation with cache + refresh.** AI analysis runs on first view (not on every session completion), is cached in the DB, and a Refresh button allows regeneration.

## Architecture

Three layers compose into one display:

```
┌─────────────────────────────────────────────────┐
│                 ANALYSIS VIEW                    │
├─────────────────────────────────────────────────┤
│  ┌───────────────────────────────────────────┐  │
│  │  MECHANICAL LAYER (instant)               │  │
│  │  Stats · Prompt History · Test Status     │  │
│  │  Parsed from events.jsonl on completion   │  │
│  └───────────────────────────────────────────┘  │
│  ┌───────────────────────────────────────────┐  │
│  │  AI LAYER (async, on first view)          │  │
│  │  Summary · Key Moments · Next Steps · ... │  │
│  │  Generated via Amplifier subprocess       │  │
│  └───────────────────────────────────────────┘  │
│  ┌───────────────────────────────────────────┐  │
│  │  LIVE DATA LAYER (on demand)              │  │
│  │  PR status · CI checks                    │  │
│  │  Fetched via `gh` CLI, never cached       │  │
│  └───────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```

- **Mechanical extraction (instant, on session completion):** Stats, prompt history, test results, files changed, git state. Parsed from events.jsonl using the existing Phase 1 event parser (extended).
- **LLM analysis (async, on first view):** Summary, section layout selection, key moments, next steps. Generated by shelling out to Amplifier subprocess with a session digest.
- **Live data fetch (deterministic, on demand):** PR status, CI check results. Fetched via `gh` CLI when the LLM includes a `changes` section with a PR link. Always fresh, never cached.

The LLM doesn't fetch git/GitHub data — it reads the session digest and tells Canvas "this was a coding session, include a Changes section." Canvas then fetches the actual PR/CI data deterministically. The LLM decides WHAT to show; deterministic pipelines provide THE DATA.

## Components

### ANALYSIS View Layout

When you click a completed session, the right panel opens the ANALYSIS view. One scrollable page:

**Header (always present, instant):**
- Session title (from Phase 1's `deriveSessionTitle`)
- Stats row: duration, prompt count, token count
- Test status badge (if tests ran): "14/14 passing" in green, or "3 failing" in red

**Prompt History (always present, instant):**
- Collapsible section, numbered list of every prompt sent
- Each entry shows the prompt text, truncated to ~100 chars with expand

**AI-Curated Sections (load async, on first view):**
- One-sentence summary at top: "Built the session panel sidebar with live status indicators..."
- Below: sections the LLM chose based on session type
- Each AI section clearly marked with subtle indicator
- "Regenerate" button at bottom to refresh AI analysis

**Loading state:** Mechanical top half renders immediately. AI sections show skeleton/placeholder that fills in when LLM responds. If LLM call fails, mechanical content stands alone — no error state needed.

### Typed Section Catalog

The LLM returns structured JSON picking from a fixed set of known section types. Canvas defines these section types, each with a known React component:

| Section Type | Renders As | Example Content |
|---|---|---|
| `summary` | One-sentence header text | "Built the session panel sidebar..." |
| `changes` | File list with line counts + PR link slot | Files modified, added, deleted |
| `key-moments` | Timestamped milestone list | "T:14 — All tests passing for first time" |
| `next-steps` | Bulleted action items | "Add error state handling to SessionStatus.tsx" |
| `decisions` | Decision + rationale pairs | "Chose SQLite over IndexedDB because..." |
| `action-items` | Checkbox list | "Follow up with team about API contract" |
| `open-questions` | Bulleted questions | "Should the cache persist across sessions?" |

The LLM picks which sections to include and fills content — but cannot invent arbitrary types. Each type has a known React component. This gives us "LLM decides structure" without unbounded rendering complexity.

If the LLM returns malformed JSON, the mechanical layer (stats, prompt history) still renders fine — the AI sections just don't appear.

### Session Digest Pipeline

When you click a completed session for the first time, Canvas builds a digest from events.jsonl.

**Extracted data (extending Phase 1's event parser):**
- All user prompts (full text — the narrative)
- Tool call names + file paths (not full output — just "read_file: src/App.tsx")
- Errors and failures (full error messages)
- Test results (pass/fail counts, failing test names)
- Files created/modified/deleted (paths only)
- Git operations (commits, pushes, PR creates)
- Timestamps for all of the above

**Token budget:** ~8–12k tokens. Most sessions fit easily. For monster sessions: keep all user prompts (narrative arc), keep full detail for last ~30 minutes, summarize older activity as counts.

## Data Flow

### LLM Integration — How Canvas Calls Amplifier

Canvas shells out to Amplifier for the AI work:

```
┌──────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────┐
│  Canvas   │    │  Temp File   │    │  Amplifier   │    │  LLM     │
│  (main    │───▶│  (JSON:      │───▶│  subprocess  │───▶│  Provider│
│  process) │    │  digest +    │    │  --oneshot    │    │  (any)   │
│           │◀───│  prompt)     │◀───│              │◀───│          │
└──────────┘    └──────────────┘    └──────────────┘    └──────────┘
     │                                                       │
     ▼                                                       │
┌──────────┐                                                 │
│  SQLite   │  Cache result, serve instantly on next view    │
│  DB       │                                                │
└──────────┘
```

1. **Canvas builds the digest** (TypeScript, in the main process) — parses events.jsonl using the existing event parser, constructs the digest object
2. **Canvas writes the digest to a temp file** — JSON, includes the digest content plus the structured prompt asking for section catalog JSON back
3. **Canvas runs Amplifier as a subprocess** — something like `amplifier run --oneshot --input digest.json` (the exact CLI interface is an implementation detail, but the point is: Amplifier receives a prompt, calls its configured provider, returns the result)
4. **Canvas reads the response** — structured JSON with summary, section selections, and content for each section
5. **Canvas caches the result** in the SQLite DB alongside the session record

**Properties:**
- **Non-blocking** — the subprocess runs async. The mechanical layer is already rendered.
- **Provider-agnostic** — Canvas never knows or cares which LLM is behind the call. Amplifier handles that.
- **Cacheable** — the result is stored once, served instantly on subsequent views. Only regenerated on explicit "Refresh."
- **Failure-safe** — if the subprocess fails (no provider configured, network down, timeout), the ANALYSIS view still works with mechanical data only. No error state needed.

### Live Data Fetch

When the LLM response includes a `changes` section with a PR reference, Canvas fetches live data:

- PR status and CI check results via `gh` CLI
- Always fresh — never cached
- Fetched on demand each time the ANALYSIS view is opened
- If `gh` is not available or the call fails, the `changes` section renders with static data only (file list without PR/CI status)

## Data Model

Extend Phase 1's session SQLite table:

### Mechanical Data (stored on session completion)

| Column | Type | Description |
|---|---|---|
| `test_status` | TEXT (JSON) | Pass/fail counts, extracted from events |
| `prompt_history` | TEXT (JSON) | Array of user prompts with timestamps |
| `files_changed` | TEXT (JSON) | Array of file paths with change type (created/modified/deleted) |
| `git_operations` | TEXT (JSON) | Array of detected git events (commits, pushes, PRs) with metadata |

### AI Data (stored on first view, or on refresh)

| Column | Type | Description |
|---|---|---|
| `analysis_json` | TEXT (JSON) | Full LLM response: summary sentence + typed sections array |
| `analysis_generated_at` | TEXT | ISO timestamp of when the analysis was generated |
| `analysis_status` | TEXT | `none` \| `loading` \| `ready` \| `failed` |

### Live Data (never stored, fetched on demand)

- PR status, CI check results — fetched via `gh` CLI when the ANALYSIS view includes a `changes` section with a PR link. Always fresh, never cached.

This means clicking a completed session is instant for the mechanical data, triggers an async LLM call if `analysis_status` is `none`, and fetches live PR/CI data if applicable. On subsequent views, everything is cached except the live data.

## Error Handling

| Failure Mode | Behavior |
|---|---|
| LLM subprocess fails (no provider, network, timeout) | Mechanical data renders normally. AI sections simply don't appear. No error UI needed. |
| LLM returns malformed JSON | Same as subprocess failure — mechanical layer stands alone. `analysis_status` set to `failed`. |
| `gh` CLI not available | `changes` section renders file list without PR/CI status badges. |
| `gh` CLI call fails (auth, network) | Same as `gh` not available — graceful degradation to static file data. |
| events.jsonl missing or corrupt | Session shows with whatever mechanical data is available. Digest builder handles partial data. |
| Token budget exceeded for digest | Digest builder truncates: keeps all user prompts, full detail for last ~30 minutes, summarizes older activity as counts. |

## Testing Strategy

- **Unit tests** for `digestBuilder.ts` — verify token budgeting, event extraction, truncation behavior
- **Unit tests** for section renderer components — verify each section type renders correctly from known JSON input
- **Unit tests** for `analysisService.ts` — mock subprocess, verify caching logic, verify failure handling
- **E2E tests** for the ANALYSIS view — click completed session, verify mechanical data renders, verify skeleton state while loading, verify AI sections appear after mock LLM response
- **E2E tests** for the Regenerate button — click refresh, verify `analysis_status` resets, verify new LLM call is triggered

## Open Questions

- **Exact Amplifier CLI interface for oneshot subprocess invocation.** The `amplifier run --oneshot --input` pattern is directional, but the exact flags and contract need to be worked out with the Amplifier CLI team.
- **Section catalog extensibility.** The initial 7 section types cover known use cases. Adding new types later is straightforward (new React component + add to the catalog), but we should validate the initial set covers 90%+ of real sessions before shipping.
- **Token count display.** Phase 1 stores `toolCallCount` but not token counts. The stats row in the ANALYSIS header wants to show token count — this may require extending the event parser to extract token usage from events.
